## 1. 개요

### 막둥이는 이런 프로그램입니다

- **AI가 결정, 사람은 감독**
  시장 분석과 매수·매도 판단은 AI가 합니다.
  사용자는 결과를 확인하고 필요할 때만 개입합니다.

- **한국 주식 전용**
  코스피·코스닥 종목을 대상으로 합니다.

- **개인 계좌 전용**
  키움증권 개인 계좌를 사용합니다. 법인 계좌·위탁 계좌는 불가합니다.

### 무엇이 다른가요

- **실시간 감시** — 보유 종목을 초 단위로 지켜보며 손절·익절 자동 실행
- **메신저 중심** — 복잡한 프로그램 창 없이 텔레그램 하나로 운영
- **자율 학습** — 매일 매매 결과를 스스로 복기해서 개선

## 2. 일상 운영

### 하루 흐름

```
  08:30 ────── 프로그램 시작 (자동 또는 수동)
  09:00 ────── 장 시작, AI 분석 활성화
  09:00~15:20  매매 진행 (주요 이벤트는 텔레그램 알림)
  15:30 ────── 장 마감, 일일 결산 자동 생성
  15:35 ────── 프로그램 자동 종료 (설정 가능)
```

### 시작 확인

프로그램이 정상 실행되면 텔레그램으로 시작 보고가 옵니다.

```
📋 Makdoongi 시작
  시각: 08:30 | 브로커: 연결됨 | 모드: 실전
```

이 메시지가 오지 않으면 프로그램이 제대로 실행되지 않은 것입니다.

### 종료 방법

- **자동 종료** — 설정된 시각(기본 15:35)에 스스로 꺼집니다.
- **수동 종료** — 실행 창을 닫거나 Ctrl+C.
- **안전 종료** — 먼저 `/emergency_stop`로 매매를 멈춘 뒤 종료 권장.

## 4-1. 직접 매수해도 자동 인식 (외부 거래 동기화)

HTS / 모바일앱 / 전화 주문 등 막둥이 외부에서 직접 매수·매도해도
자동으로 감지되어 프로그램에 반영됩니다.

### 감지 방식

- **실시간 감지**: 키움 체결 이벤트를 즉시 인식 (보통 1초 내)
- **주기 폴링**: 5분마다 계좌 잔고 변경 추가 확인 (이중 안전망)

### 알림 예시

```
🟢 외부 거래 감지 | 삼성전자 005930 | 5주 매수 @72,000원
※ HTS/앱/전화 주문으로 추정 — 프로그램이 자동 반영함
```

### 자동 처리 내용

- 매수: 새 포지션 추가 (중기/외부매수 그룹) + 평단가 자동 계산
- 매도: 기존 포지션에서 수량 감소 또는 전량 제거
- 거래 이력 자동 기록 (별도 표시: source="external")

운영자가 별도로 할 작업 없습니다.

### 프로그램 지연 체결 자동 기록

막둥이가 낸 주문이 체결 대기 시간(25초) 안에 체결 확인을 받지 못하면
일단 "매도 실패"로 처리하지만, 나중에 키움 서버에서 뒤늦게 체결이
들어오는 경우가 있습니다. (네트워크 지연·거래소 혼잡 등)

이런 지연 체결도 막둥이가 자동으로 감지해 거래 이력에 반영합니다.

```
⏰ 프로그램 주문 지연 체결 | 272210 x5 @130,800원 — 거래 기록 위임
```

덕분에 실제 체결됐지만 통계에 빠져있던 거래가 사라지지 않습니다.
(source="reconciliation"으로 구분 기록)

### 데이터 정리

막둥이 내부 포지션 기록과 브로커 실제 보유가 어긋나는 경우
(예: 장외에서 매도해서 브로커엔 없는데 파일에 남은 경우),
매도 시도 전 실보유 0주를 즉시 확인하고 파일 포지션만 조용히 정리합니다.

```
ℹ️ 데이터 정리 | 272210 x5 | 브로커 실보유 0주 확정 → 파일 포지션 제거
```

이전처럼 매도 5회 반복 실패로 5분간 오류 알림이 쌓이지 않습니다.

### 수리 큐 (비용 의식형)

크래시·코드 오류·기술 부채 중 AI 수리 비용이 클 것으로 예상되는 건은
자동으로 고치지 않고 **수리 큐에 누적**됩니다. 사용자가 여유 있을 때
몰아서 검토·승인·무시 처리할 수 있습니다.

**누적 대상:**
- 재시작 시 감지된 크래시 (Opus급 분석 필요)
- HIGH 심각도 기술 부채 (active 상태)
- 핫리로드 실패 모듈
- 시작 시 모듈 초기화 실패 (뉴스 엔진, 시작 자산 기록 등)

**자동 수리 임계값:** 추정 비용 $0.10 초과 시 자동 수리 차단 → 큐 누적
(환경변수 `REPAIR_COST_THRESHOLD_USD`로 조정 가능)

**HIGH 심각도만** 즉시 텔레그램 알림, 나머지는 조용히 누적.

**명령어:**
- `/repairs` — 대기 목록 조회 (심각도·예상비용 표시)
- `/repairs history [N]` — 처리 이력 조회 (완료/무시 포함, 기본 15건)
- `/repairs today` — 오늘 처리된 수리 요약
- `/repair <id 6자리> fix` — AI 수리 승인 (다음 AI 루프에서 시도)
- `/repair <id 6자리> skip 메모` — 무시 처리

**자동 resolve (V3.72):**
시스템 자가 복기(축 23)가 5분마다 pending 항목을 검사해 근본 원인이 해결된 것을
자동으로 `resolved` 처리합니다. 보수적 규칙만 적용:

- `crash/hang_suspected`: 최근 체크 중 healthy ≥8/10 → 자동 해소
- pending 24시간 경과 + 재현 없음 → 자연 해소

자동 resolve 발생 시 `REPAIR_AUTO_RESOLVED` 정보 레벨 이벤트로 기록되어
투명하게 추적됩니다.

**알림 예시:**
```
🔧 수리 큐 누적 a8b3c2
심각도: 🔴 HIGH
사유: 크래시 감지: hang_suspected
예상비용: $1.000 (opus)
조회: /repairs
```

## 4-2. 재시작 시 포지션 안전 보호

막둥이를 재시작해도 기존 보유 종목이 실수로 매도되지 않습니다.

### 왜 중요한가

이전 버전에서는 재시작 시 "분류되지 않은 포지션"을 자동 청산하는
로직이 일부 그룹(AI-기타 / 이전보유 / AI-초단타)을 빼먹어서,
정상 보유 종목을 전량 시장가 매도하는 사고가 발생할 수 있었습니다.
현재는 다음 2중 안전 구조로 완전 차단:

### 구조

- **블랙리스트 방식**: 자동 청산은 `AUTO_CLEANUP_GROUPS`에 **명시된 그룹만**
  대상. 기본 비어있음 — 아무것도 청산하지 않음.
- **옵트인 플래그**: `startup_auto_cleanup_enabled=False` (기본).
  실수로 True가 돼도 블랙리스트가 비어있으면 대상 0건.
- **그룹 메타 유실 방지**: `group_policy` 모듈이 단일 진실 공급원.
  재시작 시 복원된 포지션은 "이전보유" (보호 그룹)로 자동 분류.
- **저장 시 자동 보정**: `positions.json` 저장 시 빈 그룹 감지하면
  "이전보유"로 자동 보정 + 경고 로그.

### 시작 로그 예시

```
[MAIN] 시작 자동 청산 비활성 (startup_auto_cleanup_enabled=False)
```

사용자가 별도로 할 일 없습니다. 그저 안전합니다.

## 4-3. AI 판단 근거 — 과거·현재·미래 통합

AI가 매 루프마다 "과거 + 현재 + 미래" 3대 시간축 데이터를 자동으로
받아 판단합니다. 도구 호출 생략 여부와 상관없이 시스템이 능동 주입합니다.

### 수집되는 컨텍스트

**과거:**
- 손실 패턴 DB (RSI 과매수 / 추세 피크 / 주말 보유 등 반복 실수)
- 종목별 이전 거래 성과
- 최근 복기 요약 (최근 2일)
- 최근 AI 루프 판단 이력
- **연속 손절 일수** — 2일+ 시 자동 경고
- **1주 전 가격 대비** — 과열 시각화

**현재:**
- 코스피/코스닥/VIX/환율 스냅샷
- 매크로 (Fed/BOK/수익률곡선/크립토/리스크)
- 실시간 뉴스 이벤트 시그널
- 엘리트 시그널 건수

**미래:**
- 향후 N일 경제 캘린더 (High impact만)
- 실적 발표 예정 종목
- DART 공시 플래그
- 장 마감까지 남은 시간

### 200자 요약이 AI 프롬프트에 자동 주입

```
🕰️ 📉손실패턴:RSI_OVERBOUGHT×5회 | 🔥연속손실2일 |
📊코스피6,313(+1.5%)국면:상승추세 | 📰뉴스:약간매수(B2/S1) |
⏭임박:FOMC회의(04-28) | 💼실적예정3건 | ⏰마감까지3.2h
```

이 요약이 AI 판단 프롬프트에 들어가기 때문에 "도구 호출을 생략해도"
시간축 맥락을 고려한 결정이 가능합니다.

## 4-4. 시스템 자가 복기 (5분 주기)

매매 실적 복기는 `/postmortem`이 담당하지만, **매매가 아예 안 일어나는
이상 상황** (AI가 판단만 하고 주문이 안 나가는 경우 등)은 별도 체계가
5분 주기로 자동 감지합니다.

### 감지 항목 (12종, V3.72 확장)

핵심 매매:
1. **무거래 경보** — 장중 2시간+ 가동했는데 실매수 0건
2. **도구 검증 실패율** — AI 도구 호출 실패 20%+ (차단 감지)
3. **루프 정체 (초단타)** — scalp_loop 10/20분 미실행
4. **중장기 루프 정체** — midlong_loop 120/240분 미실행 *(V3.72 신규)*
5. **판단-실행 괴리** — AI는 "매수" 표명하는데 실제 체결 없음

데이터 품질:
6. **수집기 신선도** — overnight/news_events 파일 30시간+ 노쇠
7. **데이터 무결성** — 파일/DB 감사 문제 발견
8. **프롬프트 캐시 저하** — cache_read 비중 40% 미만 *(V3.72 신규)*

시스템 건전성:
9. **기술부채 매매 차단** — HIGH 부채로 매매 차단 중
10. **자가 테스트 실패** — self_diagnostics 축 실패 누적
11. **크래시 복구 빈도** — 24h 이내 복구 3회+
12. **텔레그램 알림 실패** — notifier 연속 3회+ 실패 *(V3.72 신규)*

### 심각도별 액션

- **WARNING** 🟡 — 텔레그램 알림 (같은 종류 10분 쿨다운)
- **CRITICAL** 🔴 — 텔레그램 + 수리 큐 자동 등록

### 알림 품질 개선 (V3.72)

- **심각도 상승 시 쿨다운 무시 재알림** — warning → critical 전환되면
  10분 쿨다운을 깨고 즉시 재알림
- **일일 쿨다운 자동 리셋** — 날짜가 바뀌면 모든 경보 재알림 가능
- **notifier streak 자체 추적** — 텔레그램 송신 성공/실패 연속 카운터

### 알림 예시

```
🔴 [INTEGRITY CRITICAL] NO_TRADE_DURING_ACTIVE
장중 4.2h 가동, 루프 45회 실행, 실매매 BUY 0건
```

이 알림이 뜨면 `/repairs`로 대기 수리 확인 → 우선순위대로 처리.

### 이력 조회

- `data/system_integrity_history.json` — 최근 200회 체크 기록
- 자유 입력: "시스템 체크 상태" → AI가 `get_system_integrity_status`
  도구로 즉시 재검사

## 4-5. 즉시 복기 (Phase 7) — 강제 손절/큰 손실 직후

일일 복기(`/postmortem`)는 장 마감 후 하루 묶어서 수행되지만, **강제 손절
이나 -3% 이상 손실**이 발생하면 체결 직후 즉시 원인 분석합니다.

### 동작 흐름

```
매도 체결 성공
   ↓
pnl_pct ≤ -3% 또는 "강제" 키워드 감지
   ↓
PostmortemEngine.run_instant_postmortem() 자동 호출
   ↓
1. 매수 당시 memo 조회 + hold_days 계산
2. 패턴 자동 분류 (RSI_OVERBOUGHT / ADX_PEAK / WEEKEND_HOLD /
   WEAK_SIGNAL / FORCED_STOP_LOSS)
3. 최근 30일 유사 패턴 건수 집계
4. 패턴별 개선 제안 생성
5. data/postmortems/instant/ 파일 저장
6. 텔레그램 즉시 경고 전송
```

### 알림 예시

```
🔴 즉시 복기 000660 (-5.23%)
사유: 강제 손절 발동
매수 근거: RSI 68 + ADX 52 + 반도체 모멘텀 강세
패턴: RSI_OVERBOUGHT, ADX_PEAK, FORCED_STOP_LOSS
최근 30일 유사: 8건
개선: RSI 60+ 매수 자제 / ADX 50+ 추세 피크 진입 주의 /
      강제 손절 발동 — 진입 기준 재점검
```

### 특징

- **AI 호출 0** — 순수 규칙 기반. 비용 없음.
- **즉시성** — 체결 직후 1초 내 알림.
- **매도 경로 전부 커버** — AI 자율 매도(ai_brain) + 트레이드 엔진 매도
  (position_monitor 손절/익절/트레일링) 양쪽 훅 연결
- **중복 호출 방지** — 같은 종목 60초 쿨다운으로 두 경로 동시 호출 시 이중 복기 차단
- **축 22 TemporalContext 연동** — 이후 매수 판단 시 패턴 누적이 자동 반영됨
- **파일별 저장** — 각 손실 이벤트가 독립 파일로 기록되어 이후 주간/월간
  복기에 자동 합산

## 4-6. 오탐 감사 (10분 주기)

시스템이 발령한 경보·판정이 **사후에 잘못된 것으로 드러나는 경우**를
체계적으로 감사하는 메타 레이어. 축 23이 "이상 감지"라면 축 24는
"그 감지 자체의 오탐 감지"입니다.

### 왜 필요한가 (2026-04-21 사례)

- broker "매도 실패" 로그 → 실제로는 체결 성공 (stale TR 오판)
- "외부 거래 감지" → 자체 주문이었음 (is_program_order 체크 누락)
- NO_TRADE_DURING_ACTIVE 경보 → 실제는 의도적 관망 상태

즉 **감지는 정상 작동하는 것처럼 보이지만 실제 오탐**이었던 케이스들.
축 24가 이런 패턴을 사후 감사해 감지기 품질 개선을 순환합니다.

### 5종 오탐 감사 규칙

| 규칙 | 감지 | 심각도 |
|---|---|---|
| `SELL_FAIL_FALSE_POSITIVE` | 매도 "실패" 판정 후 5분 내 실제 체결 발견 | HIGH |
| `EXTERNAL_FALSE_SELF_ORDER` | "외부 거래" 감지 주문이 `is_program_order`=True | MEDIUM |
| `NO_TRADE_ALARM_FALSE` | 무거래 경보 후 1시간 내 실제 매매 발생 | LOW |
| `VALIDATOR_CHRONIC_STRICT` | 도구 검증 실패율 30%+ 지속 (10회 이상) | MEDIUM |
| `HANG_FALSE_ALARM` | hang_suspected 후 30분간 integrity healthy 80%+ | LOW |

### 자동 액션

- **일별 누적 저장:** `data/fp_patterns/YYYY-MM-DD.json`
- **3일 내 5회+ 누적 시:** `improvement_proposal` 자동 발행 → `repair_queue`
  (수리 비용 0, 자동수리 임계 내면 즉시 수리 진행)
- **개별 오탐은 로그만** — 축 23의 INFO 스킵 정책과 연동해 텔레그램 노이즈 방지
- **카테고리별 수정 힌트** 자동 생성 (예: SELL_FAIL_FP → "_send_order pre_qty=0
  단정 거절 로직 완화")

### 조회

- 자유 입력: "오탐 통계" → AI가 `get_false_positive_summary` 도구 호출
- 패턴 파일 직접 조회: `data/fp_patterns/`

### 체계도

```
축 23 (SystemIntegrity) — 이상 상황 감지·경보 발령
    ↓
축 24 (FalsePositiveAudit) — 위 경보가 잘못됐는지 역-감사 (10분 주기)
    ↓
repair_queue 개선 제안 자동 발행 (카테고리별)
    ↓
AI/사용자 검토·수정 → 감지 품질 향상 → 축 23 재개선
```

메타 레이어이므로 **시스템 품질이 스스로 개선되는 순환**을 형성합니다.

## 4-7. 로그·메시지 아카이브 (축 26, 5분 주기) — V3.72.1 신설

시스템이 생성하는 모든 기록(파일 로그 + 텔레그램 송수신)을 **한 축이 일원화해서
생성·편집·제어·유지보수·관리**합니다.

### 왜 필요한가 (2026-04-22 사례)

- 매도 20회 반복 실패 중 로그에는 명확한 증거가 있었지만, **파일 로그를 구조화해
  참조하는 축이 없어** 축 23(system_integrity) 이 감지 규칙을 늦게 발동
- 텔레그램 송수신 메시지가 메모리 dict(`_command_stats`)에만 있어 **재시작 시
  전량 소실** — 알림 중복/누락을 사후 감사 불가
- 축 24(오탐 감사) 가 "이 경보가 중복 발송됐는가" 검증하려면 **발신 이력**이
  영속 저장돼 있어야 하는데 없었음

### 저장 체계

| 유형 | 저장소 | retention |
|---|---|---|
| 텔레그램 송신 | SQLite `telegram_messages (direction='sent')` | 30일 |
| 텔레그램 수신 | SQLite `telegram_messages (direction='received')` | 30일 |
| 파일 로그 ERROR/WARNING | SQLite `log_events` | 30일 |
| 파일 로그 원본 | `logs/makdoongi_YYYY-MM-DD.log` | 30일 (loguru) |

### 주기 작업

- 5분마다: 오늘 로그 파일 tail 파싱 → ERROR/WARNING 만 DB 에 인덱싱
- 하루 1회: retention(30일) 초과 엔트리 자동 정리
- 실시간: 텔레그램 `send()` / `_record_command()` hook 으로 매 이벤트 즉시 기록

### 다른 축과 연계

- **축 23 (system_integrity)** — `SELL_FAIL_STREAK` 같은 감지 시 로그 증거
  자동 조회해서 `evidence.recent_log_events` 에 첨부
- **축 22 (postmortem)** — 즉시 복기 실행 시 해당 종목 최근 1시간 ERROR 로그를
  `log_evidence` 필드로 자동 첨부
- **축 24 (false_positive_audit)** — 알림 중복 수 조회로 FP 판정 보조
- **향후 확장 AI 도구**: "최근 에러 요약" 질문 시 AI가 즉답 가능

### 조회 API (축간 호출)

- `query_recent_errors(hours, keyword, levels, limit)` — 로그 이벤트 필터
- `query_telegram_history(direction, hours, message_type, limit)` — 메시지 이력
- `count_recent_alert_duplicates(keyword, minutes)` — 중복 경보 카운트
- `get_summary(hours)` — 요약 통계 (AI 도구/health 보조)

### 효과

- **감지 규칙이 로그 기반 증거와 결합** → 같은 사고 재발 시 훨씬 빠른 진단
- 운영자가 특정 날짜 텔레그램 이력 완전 재구성 가능 (감사/규제 대응)
- 재시작 후에도 직전 경보/명령 흐름 이어서 조회 가능

## 4-8. 항상성 컨트롤러 (축 27, 5분 주기) — V3.72.1 신설

시스템이 스스로 "정상 상태"를 정의하고 이탈을 자동 감지·복귀시키는 축입니다.
사람의 체온·혈압이 자동 조절되듯, 포트폴리오/루프/브로커 상태가 목표 범위에서
벗어나면 자동으로 보수 모드 전환·신규 매수 차단·브로커 재연결을 수행합니다.

### 감지 대상 (off_targets)

| 지표 | 정상 범위 | 이탈 시 자동 조치 |
|---|---|---|
| 매도 실패 연속 횟수 | < 5회 | 신규 매수 차단 + 즉시 복기 발동 |
| 포지션 비중 | < safety_max_position_pct | 신규 매수 차단 |
| 일일 실현 손실 | > safety_max_daily_loss_pct | 매수 차단 + 보수 모드 |
| 브로커 연결 | 5분 내 성공 | 자동 재연결 시도 |
| 루프 정체 | 예상 주기 × 2 미만 | 워치독 트리거 + 경보 |

### AI 인지 연계

이탈 중인 지표는 **AI 프롬프트에 자동 주입**됩니다. AI 는 별도 도구 호출 없이도
항상성 상태를 매 루프마다 자동 인지하고 판단에 반영합니다.

```
🔴 항상성 이탈: sell_fail_streak, position_ratio → 신규 매수 자동 차단/보수 모드
```

/health 명령에서도 실시간 off_targets 와 최근 자동 조치 이력을 확인할 수 있습니다.

## 4-9. 성공 패턴 학습 루프 (V3.72.1 완성)

익절 5% 이상 매도 체결 시 그 거래의 **시그니처**(그룹/진입 시간대/조건 패턴)를

## 4-11. 스키마 정합성 자동 관리 (축 18 확장, V3.72.1)

코드 개선 때마다 저장 데이터 필드 의미가 바뀌는데 **레거시 값은 그대로 남아**
의미 불일치를 만듭니다(예: 포지션의 `group="AI-초단타"` 인데 `max_hold_sec=2592000(30일)`
으로 저장돼 태그가 `[장기/AI-초단타]` 로 잘못 표시). 매번 수동 핫픽스 대신 이 문제를
**능동적·유기적**으로 해결하는 체계를 도입했습니다.

### 설계 — plugin 방식 ConsistencyRule

- 소유 모듈(position_monitor / strategy_store / runtime_config / ...)이 자기 영역의
  cross-field 룰을 직접 등록 → `DataIntegrityGuard` 가 전수 검사·교정을 수행
- **각 축이 자기 데이터 책임** (유기체 원칙 — 단일 중앙 집중 X)

### 자동 실행 시점

- **핫리로드 완료 직후** (runtime_wiring 이 룰 재등록 + 감사 1회 실행)
- HOT_RELOAD_COMPLETED 이벤트 훅 (main.py 안전망)
- 축 18 주기 감사 — V3.72.1 에서 주간(168h) → 시간당(1h) 로 단축

### 교정 흐름

```
check_fn() → 위반 목록         # 예: [{ticker:049480, group:AI-초단타, current:2592000, expected:86400}]
    ↓
repair_fn()                    # 메모리(PositionMonitor) + 디스크(positions.json 원자적 쓰기) 동시 교정
    ↓
DATA_SCHEMA_MISMATCH 이벤트    # 다른 축이 반응 가능 (학습/경보 연계)
```

### AI 자발 조회

AI 도구 `audit_data_consistency` 로 AI 가 스스로 감사를 실행할 수 있습니다
(`auto_repair=false` 옵션으로 dry-run 도 가능).

### 현재 등록된 룰 (2종)

| 룰 | 소유 | 검사 | 교정 |
|---|---|---|---|
| `position_group_hold_sec_alignment` | position_monitor | AI-초단타 그룹 + max_hold_sec>86400 | memory + disk 동시 86400 로 교정 |
| `trades_buy_group_max_hold_sec_completeness` | state_manager | BUY 거래 중 group/max_hold_sec 누락 | bulk 1회 실행 + 원자적 재쓰기 (group_policy 기반 기본값) |

**향후 확장 지점**: `strategy_store` (전략 JSON 스키마), `runtime_config` (파라미터 범위),
`success_patterns` (시그니처 포맷 진화) 등. 필요 시 같은 방식으로 룰만 추가 등록하면
재시작 없이 반영됩니다.

## 4-12. 자가 감지 오탐 맥락화 (V3.72.1 감사 기반, 2026-04-22)

초기 `system_integrity` 감지기(축 23)가 **절대값만 보는** 판정 규칙 때문에

|---|---|---|
| `NO_TRADE_DURING_ACTIVE` | 장중 2h 무매수 = warning, 4h = CRITICAL | **점심시간 skip** + **안전장치 발동 시 skip** (일일 손실 한도/연속 손실/비상정지) + **포지션 만석 skip** + warning 3h / critical 6h |
| `LOOP_STALL_CRITICAL` | 장중 scalp 20분 미실행 = CRITICAL | 장마감(15:30) + **30분 grace** 추가. 16:00 이후 scalp 비실행은 정상 |
| `MIDLONG_STALL_CRITICAL` | midlong 4h 미실행 = CRITICAL | **오늘 첫 midlong 실행 후 120분 grace** 추가 + "오늘 파일 중 최신" 참조 (이전 "어제 파일이 최신" 판정 버그 수정) |

**효과**: 경보 폭주 해소, 진짜 이상 신호만 텔레그램으로 발송.

## 6. AI 비용 관리

### 기본 구조

AI(Claude)를 사용하므로 비용이 발생합니다. 막둥이는 비용 관리를 위해:

- **일일 한도** — 하루 사용할 수 있는 최대 금액 설정 (기본 $5)
- **한도 초과 시 차단** — 초과하면 AI 호출이 자동 중단되고 알림
- **비용 최적화** — 간단한 판단은 규칙으로 처리, 복잡한 것만 AI에 맡김
- **모델 분리** — 상황에 맞는 모델 선택으로 비용 절감

### 자체 계산 극대화 (V3.72)

"AI는 판단만, 계산은 전부 코드가" 원칙으로 비용을 낮추면서 품질은 향상시킵니다.

- **기술지표 사전 계산** — RSI/MACD/ADX/볼린저/거래량/추세를 코드가 미리
  계산해 표 형식으로 AI 프롬프트에 주입. AI가 `get_technical_indicators`
  도구를 호출하지 않아도 한눈에 6개 지표 확보.
- **TOOLS 정의 캐싱** — 도구 정의(약 9,000 토큰)에 5분 TTL 캐시 적용.
  반복 호출 시 토큰 재전송 안 함. 누적 75% cache hit 유지.
- **규칙 엔진 확장** — 매수/매도/시그널 규칙 엔진이 명확한 케이스는 AI 호출 없이
  즉시 결정. 애매한 케이스만 AI 위임. V3.72 기준 매도 규칙 7→10 CASE,
  매수 규칙 6→9 CASE로 확장 (예: 반등 진입, 돌파 매수, 트레일링 잠금, 추세
  붕괴 손절 등 추가).
- **동일 정보 중복 전송 제거** — DecisionContext + TemporalContext가 통합
  컨텍스트를 1회 생성해 여러 판단에 재사용 (60초 캐시).

정보량은 오히려 증가하고(더 많은 지표가 구조화돼 주입됨), AI 호출 횟수와
재전송 토큰은 감소합니다.

### 비용 확인

```
/cost    →  오늘 $X.XX / $5.00 (XX%) 사용
```

한도의 80%를 넘으면 텔레그램으로 경고가 옵니다.

## 7. 자주 생기는 상황

### 알림이 안 와요

- 텔레그램 봇을 차단하지 않았는지 확인
- 프로그램이 실행 중인지 `/status`로 확인
- 네트워크 연결 상태 확인

### 매매가 안 돼요

- 장 시간 확인 (09:00~15:20)
- `/status`로 브로커 연결 상태 확인
- 예수금이 충분한지 확인
- `/emergency_stop`이 켜져 있는지 확인 → `/resume`으로 해제

### 손실이 커요

1. **즉시 멈추기**: `/emergency_stop`
2. **상황 파악**: `/positions`, `/account`
3. **필요 시 청산**: `/close_all` (신중하게)
4. **분석**: `/postmortem`으로 당일 복기 확인

### 프로그램이 꺼졌어요

- 보통은 자동 재시작됩니다.
- 재시작 알림이 오지 않으면 컴퓨터에서 직접 다시 실행.
- 실행 파일 이름과 위치는 운영자에게 확인.

## 8. 운영 권장사항

### 일상 체크포인트

- **아침**: 프로그램 시작 보고 확인
- **점심**: `/status` 한 번 체크
- **저녁**: 일일 결산 확인, 이상 있으면 `/postmortem` 상세 확인

### 매주 확인

- `/weekly`로 주간 성과 검토
- 손실이 누적되면 AI 전략 조정 검토 (운영자 상의)

### 안전 수칙

- 전 재산을 넣지 마세요. 손실 감당 가능한 금액만 운영.
- 처음에는 소액으로 테스트하면서 패턴 파악.
- 이상한 거래가 보이면 즉시 `/emergency_stop`.
- 정기적으로 `/cost`로 AI 비용도 확인.

### 장기 운영

- 분기별로 전략 성과 점검.
- 시장 환경이 급변할 때(금리 변동, 규제 등)는 일시 중지 고려.
- 백업이 자동으로 되지만, 중요한 기록은 별도 저장 권장.

## 9. 텔레그램 알림 형식 안내

모든 알림은 모바일 한 화면에 한눈에 보이도록 컴팩트하게 설계되어 있습니다.

### 시스템 알림
```
ℹ️ 시스템 알림 | 16:07 | 핫리로드
RuntimeConfig 변경 감지 → 재로드 완료
```

### 매수/매도 실행
```
🔵 매수 | [중기] 삼성전자 005930 | 10주 × 70,000원 = 700,000원
📝 AI 매수 판단
```

### 청산 (익절/손절)
```
🔴 익절 | [중기] 삼성전자 005930 | 10주 | 70,000→72,000 🔴+2.9% +20,000원 | 15분
📝 익절 목표 도달
```

### AI 판단 보류
```
🤖 AI 판단 | 익절 보류 | 삼성전자 005930 | 70,000→72,000 🔴+2.9%
📝 추가 확인 필요 | 재판단 60초 | 강제손절 -12%
```

### 외부 거래 감지
```
🟢 외부 거래 감지 | 삼성전자 005930 | 5주 매수 @72,000원
```

### 색상 규칙 (한국 증시 표준)

- 🔴 빨강 = 수익 (양수)
- 🔵 파랑 = 손실 (음수)
- ⚪ 회색 = 변동 없음 (0)
