## 1. 개요

### 막둥이는 이런 프로그램입니다

- **AI가 결정, 사람은 감독**
  시장 분석과 매수·매도 판단은 AI가 합니다. 사용자는 결과를 확인하고 필요할 때만 개입합니다.
- **한국 주식 전용** — 코스피·코스닥 종목 대상
- **개인 계좌 전용** — 키움증권 개인 계좌. 법인/위탁 계좌는 불가.

### 무엇이 다른가요

- **실시간 감시** — 보유 종목을 초 단위로 지켜보며 손절·익절 자동 실행
- **메신저 중심** — 복잡한 프로그램 창 없이 텔레그램 하나로 운영
- **자율 학습** — 매일 매매 결과를 스스로 복기해서 개선
- **자기 인식** — 24시간 슬라이딩 윈도우로 자기 상태를 진단하고 판단에 반영

## 2. 일상 운영

### 하루 흐름

```
  08:30 ────── 프로그램 시작 (자동 또는 수동)
  09:00 ────── 장 시작, AI 분석 활성화
  09:00~15:20  매매 진행 (주요 이벤트는 텔레그램 알림)
  15:30 ────── 장 마감, 일일 결산 자동 생성
  15:35 ────── 프로그램 자동 종료 (설정 가능)
```

### 시작 확인

프로그램이 정상 실행되면 텔레그램으로 시작 보고가 옵니다.

```
📋 Makdoongi 시작
  시각: 08:30 | 브로커: 연결됨 | 모드: 실전
```

이 메시지가 오지 않으면 프로그램이 제대로 실행되지 않은 것입니다.

### 종료 방법

- **자동 종료** — 설정된 시각(기본 15:35)에 스스로 꺼집니다.
- **수동 종료** — 실행 창을 닫거나 Ctrl+C.
- **안전 종료** — 먼저 `/emergency_stop`로 매매를 멈춘 뒤 종료 권장.

## 4. 보고서와 복기

### 4-1. 일일 결산 (장 마감 후 자동)

- 오늘 거래 횟수, 수익/손실 합계
- 종목별 손익
- 내일 전략 제안
- **전략별 성과 카운트** — 전략명(ID) 없는 AI 매매는 포지션의 그룹명(`AI-초단타`·`AI-중장기` 등) 기준 카운트

### 4-2. 주간·월간 복기

- 주간/월간 누적 성과 · 승률 · 평균 수익률
- 잘된 점 / 개선점 분석

### 4-3. 매매 복기 (매 거래)

- 진입 이유 / 청산 이유
- AI의 당시 판단 근거
- 결과 대비 평가

### 4-4. 즉시 복기 — 강제 손절/큰 손실 직후

일일 복기는 장 마감 후 하루 묶어서 수행되지만, **강제 손절이나 -3% 이상 손실**이
발생하면 체결 직후 즉시 원인 분석합니다.

**동작 흐름**
```
매도 체결 성공
   ↓
pnl_pct ≤ -3% 또는 "강제" 키워드 감지
   ↓
PostmortemEngine.run_instant_postmortem() 자동 호출
   ↓
1. 매수 당시 memo 조회 + hold_days 계산
2. 패턴 자동 분류 (RSI_OVERBOUGHT / ADX_PEAK / WEEKEND_HOLD / WEAK_SIGNAL / FORCED_STOP_LOSS)
3. 최근 30일 유사 패턴 건수 집계
4. 패턴별 개선 제안 생성
5. data/postmortems/instant/ 파일 저장
6. 텔레그램 즉시 경고 전송
```

**알림 예시**
```
🔴 즉시 복기 000660 (-5.23%)
사유: 강제 손절 발동
매수 근거: RSI 68 + ADX 52 + 반도체 모멘텀 강세
패턴: RSI_OVERBOUGHT, ADX_PEAK, FORCED_STOP_LOSS
최근 30일 유사: 8건
개선: RSI 60+ 매수 자제 / ADX 50+ 추세 피크 진입 주의 / 강제 손절 발동 — 진입 기준 재점검
```

**특징**
- AI 호출 0 — 순수 규칙 기반. 비용 없음.
- 즉시성 — 체결 직후 1초 내 알림.
- 매도 경로 전부 커버 (AI 자율 매도 + 트레이드 엔진 매도 양쪽)
- 중복 호출 방지 — 같은 종목 60초 쿨다운
- 이후 매수 판단 시 패턴 누적이 자동 반영 (TemporalContext 연동)

## 5. 자가 진화 시스템

막둥이의 핵심 — "**학습·자가 인식 5단계 + Meta-cognition**" 이 완전 폐쇄 루프로 작동.

### 5-1. 학습·자가 인식 5단계

"누적 → 복기 → 학습 → 배우기 → 깨우치기" + Meta-cognition — 매 단계가 실제 작동.

| 단계 | 담당 모듈 | 메커니즘 |
|---|---|---|
| 1. 누적 | StateManager · PatternDB · `_tool_usage_stats` | trades.json · postmortems · SQLite |
| 2. 복기 | PostmortemEngine | 즉시/일일/주간/월간 자동 실행 |
| 3. 학습 | PatternDB + SuccessPatternDB + FalsePositiveAudit + CustomPatternDetector | 손실·성공·오탐·신규 발견 4종 DB |
| 4. 배우기 | DecisionContext | 과거 패턴 top3 를 매 AI 판단 프롬프트에 자동 주입 + Screener 점수 반영 |
| 5. 깨우치기 | TuningExecutor | 이벤트 기반 자율 파라미터 조정 + 3일 효과 검증 + 자동 롤백 |
| Meta-cognition | SystemSelfState | 24h 슬라이딩 윈도우 자기 진단 (5 레벨) |

**피드백 루프 3중 완전 폐쇄**
1. 손실 → PatternDB → AI 프롬프트 경고 + Screener 페널티
2. 익절 → SuccessPatternDB → Screener 보너스 점수
3. 오탐 → FP 감사 → improvement_proposal → TuningExecutor 자율 실행 → 3일 검증

### 5-2. AI 판단 근거 — 과거·현재·미래 통합

AI가 매 루프마다 "과거 + 현재 + 미래" 3대 시간축 데이터를 자동으로 받아 판단합니다.
도구 호출 생략 여부와 상관없이 시스템이 능동 주입합니다.

**과거**
- 손실 패턴 DB top3 (RSI 과매수 / 추세 피크 / 주말 보유 등 반복 실수)
- 종목별 이전 거래 성과
- 최근 복기 요약 (최근 2일)
- 최근 AI 루프 판단 이력
- 연속 손절 일수 (2일+ 시 자동 경고)
- 1주 전 가격 대비 (과열 시각화)

**현재**
- 코스피/코스닥/VIX/환율 스냅샷
- 매크로 (Fed/BOK/수익률곡선/크립토/리스크)
- 실시간 뉴스 이벤트 시그널
- 엘리트 시그널 건수

**미래**
- 향후 N일 경제 캘린더 (High impact만)
- 실적 발표 예정 종목
- DART 공시 플래그
- 장 마감까지 남은 시간

200자 요약이 AI 프롬프트에 자동 주입:
```
🕰️ 📉손실패턴:RSI_OVERBOUGHT×5회 | 🔥연속손실2일 | 📊코스피6,313(+1.5%)국면:상승추세
 | 📰뉴스:약간매수(B2/S1) | ⏭임박:FOMC회의(04-28) | 💼실적예정3건 | ⏰마감까지3.2h
```

### 5-3. 시스템 자가 복기 (축 23, 5분 주기)

매매 실적 복기는 `/postmortem`이 담당하지만, **매매가 아예 안 일어나는 이상 상황**은
별도 체계가 5분 주기로 자동 감지합니다.

**감지 항목 (12종)**

핵심 매매:
1. 무거래 경보 — 장중 3시간+ 가동했는데 실매수 0건 (맥락 인식: 점심시간/안전장치/만석 skip)
2. 도구 검증 실패율 20%+
3. 루프 정체 (초단타) — scalp_loop 10/20분 미실행 (장마감+30분 grace)
4. 중장기 루프 정체 — midlong_loop 120/240분 미실행 (오늘 첫 실행 후 120분 grace)
5. 판단-실행 괴리 — AI는 "매수" 표명하는데 실제 체결 없음

데이터 품질:
6. 수집기 신선도 — overnight/news_events 파일 30시간+ 노쇠
7. 데이터 무결성 — 파일/DB 감사 문제 발견
8. 프롬프트 캐시 저하 — cache_read 비중 40% 미만

시스템 건전성:
9. 기술부채 매매 차단 — HIGH 부채로 매매 차단 중
10. 자가 테스트 실패
11. 크래시 복구 빈도 — 24h 이내 복구 3회+
12. 텔레그램 알림 실패 — notifier 연속 3회+ 실패

**심각도별 액션**
- WARNING 🟡 — 텔레그램 알림 (같은 종류 10분 쿨다운)
- CRITICAL 🔴 — 텔레그램 + 수리 큐 자동 등록

**맥락 인식 (경보 폭주 해소)**

| 감지기 | 맥락화 |
|---|---|
| `NO_TRADE_DURING_ACTIVE` | 점심시간 skip + 안전장치 발동 시 skip + 포지션 만석 skip + 임계 3h/6h |
| `LOOP_STALL_CRITICAL` | 장마감(15:30) +30분 grace |
| `MIDLONG_STALL_CRITICAL` | 오늘 첫 midlong 실행 후 120분 grace + 오늘 파일 중 최신 참조 |

**이력 조회**
- `data/system_integrity_history.json` 최근 200회 체크
- 자유 입력: "시스템 체크 상태" → AI가 `get_system_integrity_status` 도구로 즉시 재검사

### 5-4. 오탐 감사 (축 24, 10분 주기)

시스템이 발령한 경보·판정이 **사후에 잘못된 것으로 드러나는 경우**를 감사하는 메타 레이어.
축 23이 "이상 감지"라면 축 24는 "그 감지 자체의 오탐 감지".

**5종 오탐 감사 규칙**

| 규칙 | 감지 | 심각도 |
|---|---|---|
| `SELL_FAIL_FALSE_POSITIVE` | 매도 "실패" 판정 후 5분 내 실제 체결 발견 | HIGH |
| `EXTERNAL_FALSE_SELF_ORDER` | "외부 거래" 감지 주문이 `is_program_order`=True | MEDIUM |
| `NO_TRADE_ALARM_FALSE` | 무거래 경보 후 1시간 내 실제 매매 발생 | LOW |
| `VALIDATOR_CHRONIC_STRICT` | 도구 검증 실패율 30%+ 지속 (10회 이상) | MEDIUM |
| `HANG_FALSE_ALARM` | hang_suspected 후 30분간 integrity healthy 80%+ | LOW |

**자동 액션**
- 일별 누적 저장 — `data/fp_patterns/YYYY-MM-DD.json`
- 3일 내 5회+ 누적 시 `improvement_proposal` 자동 발행 → `repair_queue`
- 카테고리별 수정 힌트 자동 생성

**체계도**
```
축 23 (SystemIntegrity) — 이상 상황 감지·경보 발령
    ↓
축 24 (FalsePositiveAudit) — 위 경보가 잘못됐는지 역-감사 (10분 주기)
    ↓
repair_queue 개선 제안 자동 발행 (카테고리별)
    ↓
TuningExecutor 자율 실행 (safety gate) → 3일 효과 검증 → 롤백
```

**시스템 품질이 스스로 개선되는 순환**을 형성합니다.

### 5-5. 항상성 컨트롤러 (축 27, 5분 주기)

시스템이 스스로 "정상 상태"를 정의하고 이탈을 자동 감지·복귀시키는 축.
사람의 체온·혈압이 자동 조절되듯, 포트폴리오/루프/브로커 상태가 목표 범위에서 벗어나면
자동으로 보수 모드 전환·신규 매수 차단·브로커 재연결을 수행합니다.

**감지 대상 (off_targets)**

| 지표 | 정상 범위 | 이탈 시 자동 조치 |
|---|---|---|
| 매도 실패 연속 횟수 | < 5회 | 신규 매수 차단 + 즉시 복기 발동 |
| 포지션 비중 | < safety_max_position_pct | 신규 매수 차단 |
| 일일 실현 손실 | > safety_max_daily_loss_pct | 매수 차단 + 보수 모드 |
| 브로커 연결 | 5분 내 성공 | 자동 재연결 시도 |
| 루프 정체 | 예상 주기 × 2 미만 | 워치독 트리거 + 경보 |

**AI 인지 연계**

이탈 중인 지표는 **AI 프롬프트에 자동 주입**됩니다. AI 는 별도 도구 호출 없이도 항상성
상태를 매 루프마다 자동 인지하고 판단에 반영합니다.

```
🔴 항상성 이탈: sell_fail_streak, position_ratio → 신규 매수 자동 차단/보수 모드
```

**상태 영속** — `data/homeostasis_state.json` 에 원자적 저장/복원. 재시작 후에도
off_targets/최근 자동 조치 이력이 보존돼 AI 프롬프트 자기인지 섹션에 즉시 반영.
`/health` 명령에서 실시간 off_targets 와 최근 자동 조치 이력 확인.

### 5-6. 성공 패턴 학습 루프

익절 5% 이상 매도 체결 시 그 거래의 **시그니처**(그룹/진입 시간대/RSI·ADX·MACD/신호강도)를
추출해 `data/success_patterns.json` 에 영속 저장합니다. 스크리너는 신규 매수 후보를
평가할 때 **Jaccard 유사도**로 과거 성공 시그니처와 매칭하여 보너스 점수를 자동 부여합니다.

- 유사 패턴 1건 매칭 → +0.2
- 2건 → +0.4 (상한)

memo 에서 RSI/ADX/MACD/신호강도 값을 정규식 파싱해 정확한 시그니처로 저장.

`/성과` 명령에서 최근 성공 패턴 N건 / 평균 수익률 요약 확인 가능. AI 도구
`get_success_patterns` 로 AI 가 자발적 조회 가능.

### 5-7. 자동 발견 손실 패턴 (CustomPatternDetector)

기존 PatternDB 의 8종 정의된 패턴 외에 **통계적으로 반복되는 신규 손실 조합**을
시스템이 스스로 발견합니다.

- 시그니처: `(group, hour_bucket, weekday_type, holding_bucket)`
- 3회 이상 반복 + 평균 -3% 이하 손실 → 자동 등록
- 매시간 자동 스캔

AI 프롬프트에 "🔬 자동 발견 손실 조합 top3" 섹션 자동 주입:
```
🔬 자동 발견 손실 조합 top3 (반복성 높음):
  · AI-초단타 그룹 / 14-16_close 진입 / thu_fri / 보유 long_gt3h → 5회, 평균 -4.2%
```

### 5-8. 자율 튜닝 (TuningExecutor)

승인 없이 자율적으로 파라미터를 조정하는 실행기. safety gate 통과 시에만 실행:

- **화이트리스트 16개 파라미터만 허용** (매매 core 보호)
- 변동폭 ±20% 이내, 24h 쿨다운
- 3일 후 효과 측정 + 악화 시 자동 롤백
- SQLite `tuning_effectiveness` 영구 감사

**이벤트 기반 자동 트리거**

| 트리거 이벤트 | 자율 조정 |
|---|---|
| `HOMEOSTASIS_OFF_TARGET` (loss_streak_high) | `ai_loop_interval_active` +20% (충동 매매 억제) |
| `HOMEOSTASIS_OFF_TARGET` (sell_fail_high) | `router_buy_rsi_overbought` +2p (보수화) |
| `FP_PROPOSAL_EMITTED` | `signal_strength_threshold` -5% (validator 과엄격 완화) |

`/tune` 명령으로 현황·이력·안전 파라미터 조회.

### 5-9. 자기 상태 진단 (SystemSelfState)

24h 슬라이딩 윈도우로 승률·손절율·연속손실·AI 신뢰도 추세·거래 빈도 변화를 계산하여
5단계 자기 판정을 내립니다.

| 판정 | 조건 |
|---|---|
| 🟢 excellent | 승률 60%+ & 손절율 <20% |
| 🔵 good | 승률 40-60% |
| 🟡 cautious | 승률 20-40% 또는 최근 추세 악화 |
| 🔴 struggling | 승률 <20% 또는 연속 손실 5회+ |
| 🟠 recovering | struggling 후 회복 징후 |

AI 프롬프트에 "🧠 24h 자기진단" 섹션 자동 주입 — "지금 나는 struggling 상태"라는
자기 인식이 매 판단에 반영됩니다.

**매시간 자동 알림** — 판정 변화 시 또는 이상 상태 진입 시 텔레그램 전송.
`/selfstate` 로 즉시 수동 조회 가능.

## 6. 인프라와 안전

### 6-1. 직접 매수 자동 인식 (외부 거래 동기화)

HTS / 모바일앱 / 전화 주문 등 막둥이 외부에서 직접 매수·매도해도 자동으로 감지되어
프로그램에 반영됩니다.

**감지 방식**
- 실시간 — 키움 체결 이벤트를 즉시 인식 (보통 1초 내)
- 주기 폴링 — 5분마다 계좌 잔고 변경 추가 확인 (이중 안전망)

**알림 예시**
```
🟢 외부 거래 감지 | 삼성전자 005930 | 5주 매수 @72,000원
※ HTS/앱/전화 주문으로 추정 — 프로그램이 자동 반영함
```

**자동 처리**
- 매수 → 새 포지션 추가 + 평단가 자동 계산
- 매도 → 기존 포지션에서 수량 감소 또는 전량 제거
- 거래 이력 자동 기록 (`source="external"`)

**프로그램 지연 체결 자동 기록**

막둥이가 낸 주문이 체결 확인을 받지 못해 "매도 실패"로 처리했지만 나중에 뒤늦게 체결이
들어오는 경우(네트워크 지연·거래소 혼잡), 이런 지연 체결도 자동 감지해 거래 이력에 반영합니다.

```
⏰ 프로그램 주문 지연 체결 | 272210 x5 @130,800원 — 거래 기록 위임
```

자기 주문 판별을 위해 SendOrder 직전 `register_own_order_intent(ticker, qty, side)`
호출 → 최근 60분 intent 매칭 → `source="program_delayed"` 로 분류하여 승률 통계 정상 반영.

**데이터 정리**

막둥이 내부 포지션 기록과 브로커 실제 보유가 어긋나는 경우, 매도 시도 전 실보유 0주를
즉시 확인하고 파일 포지션만 조용히 정리합니다.

```
ℹ️ 데이터 정리 | 272210 x5 | 브로커 실보유 0주 확정 → 파일 포지션 제거
```

**유령 포지션 이중 안전장치**
- `com_broker._ghost_sell_counter` — SELL 3중 미확인(체잔+보유+pending 모두 없음) 3회 연속
- `trade_engine` 자체 판정 — `_sell_fail_count >= 3` + 3중 미확인 키워드 탐지
- `GHOST_POSITION_DETECTED` 마커 → 메모리/파일 동시 제거 + 알림 1회

### 6-2. 수리 큐 (비용 의식형)

크래시·코드 오류·기술 부채 중 AI 수리 비용이 클 것으로 예상되는 건은 자동으로 고치지
않고 **수리 큐에 누적**됩니다.

**누적 대상**
- 재시작 시 감지된 크래시 (Opus급 분석 필요)
- HIGH 심각도 기술 부채 (active 상태)
- 핫리로드 실패 모듈
- 시작 시 모듈 초기화 실패

**자동 수리 임계** — 추정 비용 $0.10 초과 시 자동 수리 차단 → 큐 누적
(환경변수 `REPAIR_COST_THRESHOLD_USD`로 조정)

**HIGH 심각도만** 즉시 텔레그램 알림, 나머지는 조용히 누적.

**명령어**
- `/repairs` — 대기 목록 조회
- `/repairs history [N]` — 처리 이력
- `/repairs today` — 오늘 처리 요약
- `/repair <id 6자리> fix` — AI 수리 승인
- `/repair <id 6자리> skip 메모` — 무시 처리

**자동 resolve** — 시스템 자가 복기가 5분마다 pending 항목을 검사해 근본 원인이 해결된
것을 자동으로 `resolved` 처리. 보수적 규칙:
- `crash/hang_suspected`: 최근 체크 중 healthy ≥8/10 → 자동 해소
- pending 24시간 경과 + 재현 없음 → 자연 해소

### 6-3. 재시작 시 포지션 안전

막둥이를 재시작해도 기존 보유 종목이 실수로 매도되지 않습니다.

**2중 안전 구조**
- **블랙리스트 방식** — 자동 청산은 `AUTO_CLEANUP_GROUPS`에 명시된 그룹만 대상. 기본 비어있음.
- **옵트인 플래그** — `startup_auto_cleanup_enabled=False` (기본). 실수로 True가 돼도
  블랙리스트가 비어있으면 대상 0건.
- **그룹 메타 유실 방지** — `group_policy` 단일 진실 공급원. 재시작 복원 포지션은
  "이전보유" (보호 그룹)로 자동 분류.
- **저장 시 자동 보정** — `positions.json` 저장 시 빈 그룹 감지하면 "이전보유"로 자동
  보정 + 경고 로그.

**부팅 복구 4중 안전망**
1. ActionQueue 미완료 액션 복구 — 24h 이내 자동 재개 / 24h 초과 자동 거부
2. CatchupEngine — 놓친 복기/일일 카운터/야간 리포트 자동 실행
3. PositionReconciler — 브로커↔파일 3중 동기화 (유령/수량/복구)
4. CrashRecoveryEngine — 24h 내 크래시 감지 + 무결성 감사 + 강제 reconcile

### 6-4. 로그·메시지 아카이브 (축 26, 5분 주기)

시스템이 생성하는 모든 기록(파일 로그 + 텔레그램 송수신)을 한 축이 일원화해서
생성·편집·제어·유지보수·관리합니다.

**저장 체계**

| 유형 | 저장소 | retention |
|---|---|---|
| 텔레그램 송신 | SQLite `telegram_messages (direction='sent')` | 30일 |
| 텔레그램 수신 | SQLite `telegram_messages (direction='received')` | 30일 |
| 파일 로그 ERROR/WARNING | SQLite `log_events` | 30일 |
| 파일 로그 원본 | `logs/makdoongi_YYYY-MM-DD.log` | 30일 (loguru) |

**주기 작업**
- 5분마다: 오늘 로그 파일 tail 파싱 → ERROR/WARNING 만 DB 에 인덱싱
- 하루 1회: retention(30일) 초과 엔트리 자동 정리
- 실시간: 텔레그램 송수신 hook 으로 매 이벤트 즉시 기록

**다른 축과 연계**
- 축 23 (system_integrity) — 감지 시 로그 증거 자동 조회해서 evidence 에 첨부
- 축 22 (postmortem) — 즉시 복기 실행 시 해당 종목 최근 1시간 ERROR 로그 첨부
- 축 24 (false_positive_audit) — 알림 중복 수 조회로 FP 판정 보조

**효과**
- 감지 규칙이 로그 기반 증거와 결합 → 같은 사고 재발 시 빠른 진단
- 운영자가 특정 날짜 텔레그램 이력 완전 재구성 가능
- 재시작 후에도 직전 경보/명령 흐름 이어서 조회 가능

### 6-5. 스키마 정합성 자동 관리 (축 18 확장)

코드 개선 때마다 저장 데이터 필드 의미가 바뀌는데 **레거시 값은 그대로 남아** 의미
불일치를 만듭니다(예: `group="AI-초단타"` 인데 `max_hold_sec=2592000(30일)` 으로 저장돼
태그가 `[장기/AI-초단타]` 로 잘못 표시). 이 문제를 **능동적·유기적**으로 해결.

**설계 — plugin 방식 ConsistencyRule**

- 소유 모듈(position_monitor / strategy_store / runtime_config 등)이 자기 영역의
  cross-field 룰을 직접 등록 → `DataIntegrityGuard` 가 전수 검사·교정
- 각 축이 자기 데이터 책임 (단일 중앙 집중 X)

**자동 실행 시점**
- 핫리로드 완료 직후 (runtime_wiring 룰 재등록 + 감사 1회 실행)
- `HOT_RELOAD_COMPLETED` 이벤트 훅
- 축 18 주기 감사 (1시간)

**핫리로드 ↔ 메모리 동기화 3층 안전장치**
1. `PositionMonitor.reload_from_file()` — 파일 기준 메모리 재동기화 (유령 자동 제거)
2. `brain/hot_reload.py` — hot_reload 감지 시 reloaded 리스트가 비어있지 않으면 항상 호출
3. `brain/runtime_wiring.py` — ConsistencyRule 감사 직후에도 호출

**AI 자발 조회** — AI 도구 `audit_data_consistency` (auto_repair 옵션)

**등록된 룰 3종**

| 룰 | 소유 | 검사 | 교정 |
|---|---|---|---|
| `position_group_hold_sec_alignment` | position_monitor | AI-초단타 + max_hold_sec>86400 | memory + disk 동시 86400 로 |
| `position_group_restore_from_trade_history` | position_monitor | group="이전보유/빈/AI-기타" + trades 최근 BUY 에 구체 AI-XX | trades 의 실제 group 으로 복원 + max_hold_sec 재설정 |
| `trades_buy_group_max_hold_sec_completeness` | state_manager | BUY 거래 중 group/max_hold_sec 누락 | bulk 1회 실행 + 원자적 재쓰기 |

### 6-6. 자투리 포지션 관리

AI 가 예수금 부족 또는 그룹 배분 말미에 **8천~4만원대 자투리** 매수를 반복하던 문제를
양방향으로 해결.

**신규 매수 차단** (`min_order_amount`, 기본 500,000원)
- `quantity × price` 가 이 값 미만이면 safety_layer 에서 BUY 차단
- AI 도구 `buy_stock` description 에 명시 → AI 가 자발적으로 회피

**기존 자투리 자동 청산** (`tiny_liquidation_enabled`)
- `position_monitor.on_tick` 6단계로 동작 (강제손절/손절/익절/트레일링/기간 다음)
- **자투리(cost < min_order_amount)** 중 **pnl ≥ threshold** 이면 즉시 시장가 매도
- `tiny_liquidation_threshold_pct` (기본 0.0 = 본전 이상)

**운영 예**
```
/set_config tiny_liquidation_enabled true          # 청산 모드 활성
/set_config tiny_liquidation_threshold_pct -0.5    # 0.5% 손실까지 허용
/set_config min_order_amount 300000                # 주문 하한 30만원으로 완화
/set_config min_order_amount 0                     # 주문 하한 비활성
```

### 6-7. AI 자기 분류·통계

AI 자기 이해와 운영자 가시성을 위해 내부 자원을 명시적으로 분류.

- **107개 AI 도구** — 22개 카테고리 (market_data/trade/analysis/system 등)
- **143개 런타임 파라미터** — 14개 그룹 (ai_loop/trade/safety/router 등)
- **메타 축 (tier 5)** — 기능별 meta_group 부여 (self_healing/audit/infra/archive)

신규 도구 `get_tool_usage_stats` — AI 의 도구 사용 누적 통계(호출수/오류율/최근 시각)를
AI 자신이 조회하여 빈도가 낮은 도구를 의식적으로 활용.

- **강제 손절** — 더 큰 하락 % 도달 시 무조건 청산
- **익절** — 설정 % 상승 시 AI 판단으로 청산
- **트레일링 스탑** — 고점 대비 일정 % 하락 시 자동 청산

### 계좌 보호

- **일일 거래 횟수 한도** — 과도한 매매 방지
- **예수금 비중 제한** — 한 종목에 몰빵 방지
- **일일 손실 상한** — 정해진 손실액에 도달하면 당일 매매 중단

### 장외 보호

- 장 시간(09:00~15:20) 외에는 매매하지 않습니다.
- 공휴일, 임시 휴장일 자동 인식.

## 8. AI 비용 관리

### 기본 구조

AI(Claude)를 사용하므로 비용이 발생합니다. 막둥이는 비용 관리를 위해:

- **일일 한도** — 하루 사용할 수 있는 최대 금액 설정 (기본 $5)
- **한도 초과 시 차단** — 초과하면 AI 호출이 자동 중단되고 알림
- **비용 최적화** — 간단한 판단은 규칙으로 처리, 복잡한 것만 AI에 맡김
- **모델 분리** — 상황에 맞는 모델 선택 (Opus / Sonnet / Haiku)

### 자체 계산 극대화

"AI는 판단만, 계산은 전부 코드가" 원칙으로 비용을 낮추면서 품질은 향상시킵니다.

- **기술지표 사전 계산** — RSI/MACD/ADX/볼린저/거래량/추세를 코드가 미리 계산해 표
  형식으로 AI 프롬프트에 주입
- **TOOLS 정의 캐싱** — 도구 정의(약 9,000 토큰)에 5분 TTL 캐시 적용. 누적 75% cache hit
- **규칙 엔진 확장** — 매수/매도/시그널 규칙 엔진이 명확한 케이스는 AI 호출 없이 즉시 결정
- **동일 정보 중복 전송 제거** — DecisionContext + TemporalContext가 통합 컨텍스트를
  1회 생성해 여러 판단에 재사용 (60초 캐시)

### 비용 확인

```
/cost    →  오늘 $X.XX / $5.00 (XX%) 사용
```

한도의 80%를 넘으면 텔레그램으로 경고가 옵니다.

## 9. 자주 생기는 상황

### 알림이 안 와요

- 텔레그램 봇을 차단하지 않았는지 확인
- 프로그램이 실행 중인지 `/status`로 확인
- 네트워크 연결 상태 확인

### 매매가 안 돼요

- 장 시간 확인 (09:00~15:20)
- `/status`로 브로커 연결 상태 확인
- 예수금이 충분한지 확인
- `/emergency_stop`이 켜져 있는지 확인 → `/resume`으로 해제
- `/selfstate` 로 자기 진단 확인 (struggling / cautious 상태면 자동 보수화 중)

### 손실이 커요

1. 즉시 멈추기: `/emergency_stop`
2. 상황 파악: `/positions`, `/account`
3. 필요 시 청산: `/close_all` (신중하게)
4. 분석: `/postmortem` · `/selfstate` 로 당일 복기 확인

### 프로그램이 꺼졌어요

- 보통은 자동 재시작됩니다.
- 재시작 알림이 오지 않으면 컴퓨터에서 직접 다시 실행.
- 재시작 후 `/selfstate` 로 이전 상태 복원 확인.

## 10. 운영 권장사항

### 일상 체크포인트

- **아침** — 프로그램 시작 보고 확인
- **점심** — `/status` 한 번 체크
- **저녁** — 일일 결산 확인, 이상 있으면 `/postmortem` 상세 확인

### 매주 확인

- `/weekly` 주간 성과 검토
- 손실이 누적되면 AI 전략 조정 검토 (운영자 상의)

### 안전 수칙

- 전 재산을 넣지 마세요. 손실 감당 가능한 금액만 운영.
- 처음에는 소액으로 테스트하면서 패턴 파악.
- 이상한 거래가 보이면 즉시 `/emergency_stop`.
- 정기적으로 `/cost` 로 AI 비용도 확인.

### 장기 운영

- 분기별로 전략 성과 점검.
- 시장 환경이 급변할 때(금리 변동, 규제 등)는 일시 중지 고려.
- 백업이 자동으로 되지만, 중요한 기록은 별도 저장 권장.

## 11. 텔레그램 알림 형식

모든 알림은 모바일 한 화면에 한눈에 보이도록 컴팩트하게 설계되어 있습니다.

### 시스템 알림
```
ℹ️ 시스템 알림 | 16:07 | 핫리로드
RuntimeConfig 변경 감지 → 재로드 완료
```

### 매수/매도 실행
```
🔵 매수 | [중기] 삼성전자 005930 | 10주 × 70,000원 = 700,000원
📝 AI 매수 판단
```

### 청산 (익절/손절)
```
🔴 익절 | [중기] 삼성전자 005930 | 10주 | 70,000→72,000 🔴+2.9% +20,000원 | 15분
📝 익절 목표 도달
```

### AI 판단 보류
```
🤖 AI 판단 | 익절 보류 | 삼성전자 005930 | 70,000→72,000 🔴+2.9%
📝 추가 확인 필요 | 재판단 60초 | 강제손절 -12%
```

### 외부 거래 감지
```
🟢 외부 거래 감지 | 삼성전자 005930 | 5주 매수 @72,000원
```

### 자기 진단 (매시간)
```
🔴 시스템 자기진단: STRUGGLING
  · 24h 매매 20건 (매수 12/매도 3)
  · 승률 15% / 손절율 65%
  · 평균 손익 -2.30% / 연속손실 4
  · 추세: declining / 빈도: rising
```

### 색상 규칙 (한국 증시 표준)

- 🔴 빨강 = 수익 (양수)
- 🔵 파랑 = 손실 (음수)
- ⚪ 회색 = 변동 없음 (0)
